stuct字节对齐

在C和C++中，struct的成员在内存中的排序方式受到字节对齐的影响，struct的字节对齐通常以结构体的成员中具有最大对齐需求的成员为基准。例如，如果一个struct中包含int和char类型的成员，那么由于int的字节对齐需求通常较大，因此整个结构体将按照int的字节大小进行字节对齐。

在我的理解中，字节对齐可以简单看作为：将连续的内存按照对齐值（如：x字节）分割为n个内存块（n为正整数），每个内存块的大小为对齐值（x字节）的大小，结构体中的成员会依次放入内存块中，若内存块中剩余的空间不足完整的放下当前成员，则将当前成员放置在下一个内存块的起始位置。

通过具体的实验验证这一特性：

  1. 假设由如下三个结构体：
struct struct_1 {
    uint8_t a;   // uint8_t占用一个字节
    uint16_t b;  // uint16_t占用两个字节
    uint32_t c;  // uint32_t占用四个字节
};

struct struct_2 {
    uint16_t a;
    uint16_t b;
    uint32_t c;
};

struct struct_3 {
    uint16_t a;
    uint32_t b;
    uint16_t c;
};
  1. 分别输出三个结构体占用字节数
printf("struct_1 size: %d\n", sizeof(struct struct_1));  // struct_1 size: 8
printf("struct_2 size: %d\n", sizeof(struct struct_2));  // struct_2 size: 8
printf("struct_3 size: %d\n", sizeof(struct struct_3));  // struct_3 size: 12

受到字节对齐的影响，结构体中成员的类型与顺序都会影响到最终整个结构体占用的内存大小。

  1. 结构体在内存中实际的分布

  • struct1中uint32_t的字节对齐需求最大，因此struct1以4字节对齐。
  • 由于uint8_t与uint16_t分别占用1字节与2字节，且两个成员总大小小于4字节，因此两个成员处于内存中的前3个字节。
  • 由于uint32_t大小为4字节，第一个内存块无法完整放入uint32_t，因此uint32_t存入下一个内存块中。最终sturct1占用8个字节。
  • struct2与struct3同理。

struct字节对齐引发的问题

  • 不合理的成员顺序，可能会造成大量的内存空间浪费。
  • 若使用内存拷贝对结构体进行赋值，若结构体内存不是紧凑的，即有部分填充字节，则可能无法获得正确的拷贝结果。
    • 如sturct1中想要赋值的结果为a = 0x00, b = 0x0102, c = 0x03040506，而实际结果为a = 0x00, b = 0x0203, c = 0x04050607。
  • 注意同一内存块中的不同的成员变量并不一定是紧凑的顺序，即用来占位的内存并不一定全部都在内存块的结尾。（参考struct1的内存分布情况）
  • 使用内存拷贝的方式赋值时，需要注意大小端存储的问题。
    • 如struct1中从内存角度上看为a = 0x00, b = 0x0203, c = 0x04050607，而若在程序中使用printf输出的结果可能为a = 0x00, b = 0x0302, c = 0x07060504。

struct字节对齐的优势

  • 两个同类型的结构体变量可以直接赋值。
  • 提高访问速度。
  • 减少内存浪费。
